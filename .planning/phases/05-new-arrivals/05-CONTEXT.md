# Phase 2: New Arrivals Detection — Context

**Gathered:** 2026-05-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Фаза 2 добавляет детектор новых вариантов в выдаче: если появилась новая дата вылета или новый отель/номер, бот выделяет это отдельным алертом в Telegram-отчёте.

**Что уже есть:**
- `format_changes` ловит `🆕 Новое предложение` для полностью новых offer-identity
- Но все изменения в одной куче, и на первом запуске подавляются
- Новые даты специально не выделены

**Что добавится:**
- Отдельный алерт `📅 *Новые даты вылета*` когда появляется дата, которой не было
- Отдельный алерт `🏩 *Новые отели/номера*` когда на существующей дате появляется новый вариант
- Не срабатывает при пустом предыдущем снепшоте

</domain>

<decisions>
## Implementation Decisions

### New dates detection (NEW-01)
- **D-01:** Извлечь уникальные `departure_date` из текущего и предыдущего снепшотов
- **D-02:** Исключить external-ключи (где departure_date == "external") — они не даты
- **D-03:** Разница = даты в текущем, которых нет в предыдущем
- **D-04:** Если предыдущий снепшот пуст — не алертить (первый запуск)
- **D-05:** Алерт содержит список новых дат и для каждой — лучшую цену по всем ночам

### New hotels/rooms detection (NEW-02)
- **D-06:** Для дат, которые есть в обоих снепшотах, сравнить набор hotel_option_id/room
- **D-07:** Разница = варианты (identity suffix), которые есть в текущем, но не в предыдущем
- **D-08:** Алерт содержит дату, отель/номер и цену

### Integration
- **D-09:** `format_new_arrivals(previous, current) -> (str | None, str | None)` возвращает кортеж (даты, отели)
- **D-10:** Добавить оба алерта в секцию алертов `run_check`
- **D-11:** Не создавать отдельную клавиатуру/команду — это автоматический алерт при каждой проверке

</decisions>
